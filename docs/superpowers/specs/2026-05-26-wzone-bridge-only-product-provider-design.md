# WZone Bridge-Only Product Provider Design

## Objective

Connect the serverless DealsRKY application to the existing WordPress/WZone
installation as its only external product-data provider. The application must
not fetch product pages from Amazon and must not fall back to RapidAPI when the
bridge is configured.

DealsRKY does not display a fixed product price. The bridge integration only
synchronizes saved product content such as title, images, category,
description, and features. Existing outbound `Go to Amazon` links remain
affiliate-tagged links generated by DealsRKY so that the shopper checks current
price, availability, delivery, and checkout on Amazon.

The WordPress installation remains on the current cPanel host for this phase.
Moving WordPress, WooCommerce, WZone licensing, settings, media, and product
records to a VPS is a separate migration project and is not required to make
the data flow work.

## Verified WZone Capabilities

The uploaded plugin source is WZone version `14.1.03`.

- The `NO API Keys Import` module loads WZone's bundled
  `amazon-scraper` parser and accepts `page_content`, `asin`, `country`, and
  variations before importing a WooCommerce product.
- The module's browser code obtains product-page content through its
  browser/extension workflow and then submits that content to WordPress.
- The `NO API Keys Sync Widget` module makes its own Amazon request for
  existing imported products and updates title and price only.
- WZone has an ASIN-only server-side lookup method through its PA-API helper.
  That path requires usable Amazon API credentials.

Therefore the installed no-key code does not expose a verified backend
operation that accepts only a previously unknown ASIN and returns complete
product data. This phase must not create an undocumented hidden scraper to
simulate that behavior.

## Supported Behavior

### Existing WZone Products

For an ASIN already imported into WordPress/WooCommerce by WZone:

1. The serverless site posts `asin` and `marketplace` to the authenticated
   WordPress bridge.
2. The bridge locates the WZone-managed WooCommerce product by WZone ASIN
   metadata, verifying the requested marketplace against stored import-country
   metadata when it exists.
3. The bridge reads saved WooCommerce/WZone data and returns normalized JSON.
4. The serverless site creates or updates its D1 product record from that JSON.

This path covers initial serverless ingestion of existing WZone products,
updates, enrich operations, portal-created product references, and
sheet-driven references.

### Product Refresh

The bridge returns the latest content values already saved by WZone. When an
operator re-imports or otherwise updates product content using WZone's
supported import workflow, a subsequent DealsRKY refresh re-reads those
stored values. The bridge does not call private WZone sync methods or
reproduce their network behavior.

WZone's no-key scheduled sync is not a content refresh mechanism for this
integration: the inspected implementation updates title and price for already
managed products, while DealsRKY does not consume price and requires images,
description, and features.

### Pending ASINs

When a new or existing DealsRKY ASIN is absent from the WZone-managed
WooCommerce catalog, the bridge returns a structured
`not_imported_in_wzone` response. DealsRKY keeps the ASIN eligible for a later
retry and shows an operator-facing `Pending WZone import` state rather than
claiming that product content was fetched.

The operator imports that ASIN through WZone's existing No API
Import/browser-extension workflow. After WZone saves the product, a retry or
refresh imports its stored full content into DealsRKY.

Automatic import of an unknown ASIN can be added only after one of these is
available:

- working WZone PA-API credentials for its ASIN lookup/import path, or
- a documented WZone/vendor automation endpoint that accepts an ASIN without
  the serverless application supplying scraped page content.

## WordPress Bridge

Replace the existing `wzone-bridge.php` implementation with a small
authenticated REST adapter.

### Endpoints

- `GET /wp-json/wzone-bridge/v1/health`
  - Public status endpoint.
  - Identifies whether WooCommerce and WZone are active.
  - Does not expose secrets or product data.
- `POST /wp-json/wzone-bridge/v1/fetch-product`
  - Requires `X-Bridge-Key`.
  - Validates ASIN and marketplace.
  - Reads one imported WZone/WooCommerce product and returns normalized data.
  - Never performs an Amazon request.

### Normalized Response

Successful response:

```json
{
  "success": true,
  "source": "wzone_catalog",
  "data": {
    "title": "Product title",
    "imageUrl": "https://...",
    "category": "Category or null",
    "description": "Description or null",
    "features": [],
    "productImages": [],
    "aplusImages": []
  }
}
```

Not imported response:

```json
{
  "success": false,
  "code": "not_imported_in_wzone",
  "error": "Product is not imported in WZone yet.",
  "asin": "B012345678",
  "marketplace": "US"
}
```

### Data Extraction

The bridge reads:

- `_amzASIN` and `_amzaff_prodid` for ASIN lookup compatibility.
- `_amzaff_country` where present for marketplace validation.
- WooCommerce product title, description, short description, categories,
  featured image, and gallery images.
- Stored WZone metadata only where it is already present and safely mappable
  to the normalized response.

If a non-required optional field is missing, the bridge returns an empty value
instead of trying to obtain it from Amazon.

### Security

- Continue using a separate high-entropy `X-Bridge-Key`.
- Compare keys with `hash_equals`.
- Do not expose a public debug endpoint that scans WZone PHP code or leaks
  server paths.
- Do not expose admin key retrieval as part of the production API surface.
- Validate ASIN and supported marketplace before querying WordPress data.

## Serverless Application

The WordPress bridge becomes the configured product provider:

- New serverless fetch operations call the bridge.
- Existing product refresh operations call the bridge without requiring
  `AMAZON_API_KEY`.
- Enrich flows call the bridge without requiring `AMAZON_API_KEY`.
- Portal, redirect, sheet import, sheet-control, and dynamic-link creation
  paths pass the bridge configuration whenever they can create or refresh
  product records.
- When bridge credentials are present, no RapidAPI fallback is attempted.
- If neither bridge nor another explicitly chosen future provider is
  configured, the operation fails with a configuration error.
- Missing WZone imports are surfaced as pending imports and do not create
  placeholder fetched product content.
- Bridge responses do not supply or overwrite product prices; the storefront
  continues to send shoppers to affiliate-tagged Amazon links for live price
  and purchase details.

RapidAPI code may remain temporarily for unrelated deployments, but it cannot
be an implicit fallback in this WordPress/WZone-configured deployment.

## Error Handling

The serverless application maps bridge responses as follows:

- `401` or `403`: bridge key/configuration error.
- `404` with `not_imported_in_wzone`: prompt the operator to import the
  product in WZone and retry.
- `5xx` or network timeout: WordPress/WZone bridge currently unavailable;
  permit retry.
- Invalid JSON or missing title: invalid bridge response; do not save
  placeholder product content as fetched data.

## Verification

Implementation verification must cover:

- Static check that the replacement bridge contains no `wp_remote_get` or
  `wp_remote_post` request to Amazon and no scraper/parser implementation.
- Bridge behavior on an imported WZone product and a missing ASIN after the
  PHP file is uploaded to WordPress.
- Serverless unit/API tests for bridge success, bridge authentication failure,
  pending WZone import, and bridge-only no-fallback behavior.
- Verification that existing affiliate-tagged outbound Amazon links are
  unchanged and that the bridge does not populate displayed price content.
- Serverless typecheck and affected test suite.
- Live end-to-end fetch of one already imported WZone ASIN after bridge
  deployment and serverless secret configuration.

## Deployment Responsibility

The bridge PHP replacement will be produced in the repository at
`wp-wzone-bridge/wzone-bridge.php`. Because WordPress is hosted in cPanel and
is not accessible from the local workspace, the user will replace the active
plugin PHP content in cPanel and activate/reload the plugin if required.
Serverless code changes and their local verification will be performed in this
workspace.
